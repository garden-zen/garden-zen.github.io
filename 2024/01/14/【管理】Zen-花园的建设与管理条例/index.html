论博客管理的必要性

   一个系统，在孤立的条件下，总是会自发地从有序趋于无序，此所谓“熵增原理”。为了保持系统相对的有序性，集体需要凝聚共识，社会需要维持秩序，文件的读写需要加锁，博客平台当然也需要管理。一个博客平台，倘若缺少管理，随着内容的增多，整体往往也趋于混乱。为此，我们认为，在 Zen 花园的建设中，从多方面，尤其是内容方面，加强对博客的管理，是必要的，是迫在眉睫的。这有利于维持博客形式的一致性与内容的有序性，有利于减少歧义，避免原则的模糊不清，抑制灰色地带的出现。

   基于上述观念，我们推出了如下的 《Zen 花园建设与管理条例》，别名《Zen 花园园丁手册》。

条例细则

  1. Markdown 文件的 Front Matter 应当尽量维护 标签（tags）与 分类（categories）的 一致性。
  2. Markdown 文件的 Front Matter 原则上必须设置 auto_excerpt: true（否则，默认显示全文），描述（description）为必填项。
  3. 关于文章目录设置。较长的博客文章应当添加目录，方便阅读查看。具体的做法是，在 Markdown 文件的 Front Matter 中设置 toc: true。
  4. 关于文章的置顶。置顶的文章不宜过多，最好控制在 \(3\) 篇左右，最多不应超过 \(10\) 篇。（这两个数字定的比较随机，不排除后面对它们作较大的改动）。置顶的博文可以是：关于作者和 Zen 花园的介绍（如果有的话）、对他人有较大帮助的文章、虽然不满足前两个特征但是有必要置顶的文章（比如，出于作者的虚荣心）。
  5. 关于标题格式。除极个别例外情况，所有博客文章的标题都应该以“【】”打头，方括号内填写一个最贴近文章的关键词，最好能体现文章内容的性质，同时，关键词不宜过长，以 \(1 - 4\) 个字为宜。
  6. 关于 Markdown 撰写正文时的使用规范。使用 Markdown 撰写算法题解相关的博文时，原则上要尽力达到 洛谷题解规范 的标准（实践中，部分例外是允许的），若无特殊情况，其他博文的 Markdown 格式也应遵循该规范。
  7. 关于上一条中 洛谷题解规范 的例外说明。（允许例外出现的根本目的是，确保博文写作能够使用更加合适的方法更好地表意，在个人习惯、能力与规范之间达到平衡点，而不必完全拘泥于一种形式）
    • 矩阵。矩阵的排版可以根据个人习惯使用 pmatrix，可以不使用规范中推荐的 bmatrix，但是在同一博文中二者不能混用。
    • 位运算。可以使用两种符号系统（双轨制）。第一种 是，用 \(\&\)，\(|\)，\(\oplus\)，\(\gg\)，\(\ll\) 分别表示与、或、异或、右移、左移，这种写法在 TAOCP（卷 4A）和 Hacker's Delight 中均有所体现，特别是博文中需要用到左移、右移符号时，可以使用这一套符号系统；第二种 是洛谷题解规范中推荐使用的符号，即，用 \(\operatorname{and}\)，\(\operatorname{or}\)，\(\operatorname{xor}\) 分别表示与、或、异或，当位运不涉及左移或右移符号时，可以选择这一套符号体系。同样地，在同一博文中，这两种符号系统不能混用。
    • 关于赋值。DP 中的状态转移方程可以不使用箭头符号 \(\to\) 和 \(\gets\) 表示赋值，而使用 \(=\)，《算法导论》中采用的正是这种做法。除非在语境中表示赋值的含义非常明确，一般而言，不必刻意使用箭头符号。（洛谷规范中推荐使用箭头符号 \(\to\) 和 \(\gets\)，而非代码中使用的 \(=\)）
  8. 关于一些未来可能会出现的内容的分类说明。例如：C++、Rust、操作系统、计算机网络、数据库、编译原理等相关的博客，它们更多地是为现实中的技术服务的，可以将其归为“技术”一类，算法虽然客观上也为技术服务，但是它的理论性更强，可以将其单独归为“算法”大类，数据结构类似，由于它常常与算法相提并论，也可将其归为“算法”一类。
  9. 关于超链接。除去 Hexo 自动生成的目录以外，由于 Markdown 超链接暗蓝的颜色在博文中不容易被注意到的缘故，应当使用 HTML 中的下划线标签 <u> 超链接 </u> 或其他实现方案为它添加下划线，以引起注意。
  10. 关于段首空格。算法和数学相关的博客，每一段段首可以不加空格，与其他博客平台的博文保持风格的一致性（为了凸显论述的层次，有的地方，比如一些 note 中，也可以酌情加上，一般不必加）；其余博文原则上每段段首应当空两格，实现方案推荐在每段开头加上 &emsp;&emsp;。
  11. 关于句号。算法和数学相关的博客中，使用实心圆点 .，其余博文原则上应当使用 。。
  12. 在博文中，应当尽量对有可能影响博文观感和质量的部分给予说明，方便后期的阅读与完善。比如，自己不理解的部分、模糊不清的部分、内容存疑的部分、被省略的部分、引用的部分等。
  13. 撰写博文之前，应当对题材有所选择，有所甄别，不是所有的学习过的知识都适合用博客的形式来表达。
  14. 不能为了写博客而写博客。除非特殊需要，博文的主体内容应当是自己已经理解的，可以接受的内容。目前来说，尤其是数学和算法相关的博文，应当特别注意这一点，尚不理解的部分应该及时注明。一般地，在阅读过至少一遍书本，对内容有了大致的把握与了解过后方可将其写成博文。
  15. 保留修改本条例的权利。应当根据实际的 Zen 花园建设情况与需要适时地调整与更新此条例，切勿将其视作一成不变的规则，墨守成规。倘若对某项已有的细则作出重大的调整，应当在此注明。
